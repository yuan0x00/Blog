1. Activity 启动流程 ​

题目：请详细描述 Activity 的启动流程，从 startActivity() 到 onResume() 的完整过程

考察点：

  • AMS、ActivityThread、Instrumentation 的作用
  • 应用进程与系统服务的交互
  • 生命周期回调的时机

参考答案：

1. 应用进程调用 startActivity()
2. 通过 Binder 调用到 AMS
3. AMS 检查权限、创建 ActivityRecord、找到目标进程
4. 如果目标进程不存在，通过 zygote fork 新进程
5. 目标进程启动 ActivityThread，调用 main()
6. 创建 Application，attachBaseContext()
7. 创建 Activity 实例，调用 onCreate()
8. 依次执行 onStart()、onResume()
9. 完成 UI 绘制并显示

3. Binder 机制原理 ​

题目：解释 Binder 的工作原理，为什么 Android 选择 Binder 而不是传统的 IPC

参考答案：

  • 一次拷贝：通过内存映射，数据只需拷贝一次
  • 安全性：基于 C/S 架构，身份验证完善
  • 性能：相比 Socket、管道等性能更好
  • 面向对象：可以像调用本地方法一样调用远程方法

7. MVVM 与 MVP 对比 ​

题目：比较 MVVM 和 MVP 架构的优缺点，以及在 Android 中的具体实现

参考答案：

kotlin
// MVVM 示例
class UserViewModel : ViewModel() {
    private val _users = MutableLiveData<List<User>>()
    val users: LiveData<List<User>> = _users
    
    fun loadUsers() {
        viewModelScope.launch {
            _users.value = repository.getUsers()
        }
    }
}

// MVP 示例
class UserPresenter(
    private val view: UserContract.View,
    private val repository: UserRepository
) {
    fun loadUsers() {
        view.showLoading()
        repository.getUsers { users ->
            view.hideLoading()
            view.showUsers(users)
        }
    }
}

// 对比：
// MVVM 优点：数据绑定，代码简洁，易于测试
// MVVM 缺点：调试困难，过度绑定可能导致性能问题
// MVP 优点：职责清晰，易于理解，接口明确
// MVP 缺点：接口膨胀，样板代码多

9. Flutter 原理 ​

题目：解释 Flutter 的渲染原理，与原生 Android 的差异

参考答案：

Flutter 渲染流程：
1. Widget Tree → 描述 UI 结构
2. Element Tree → 管理 Widget 生命周期  
3. RenderObject Tree → 负责布局和绘制

与原生 Android 差异：
- 自带渲染引擎 Skia，不依赖系统控件
- 响应式编程模式
- 一致的跨平台体验
- 更好的性能，60fps 流畅度

10. 混合开发技术选型 ​

题目：在什么场景下选择 Flutter/RN/原生开发？各自的优缺点是什么？

参考答案：

Flutter 适用场景：
- 追求高性能和流畅体验
- 需要高度自定义 UI
- 跨平台一致性要求高
- 团队技术栈偏向 Dart

React Native 适用场景：
- 已有 React 技术栈
- 快速迭代需求
- 丰富的第三方库生态
- 热更新需求强烈

原生开发适用场景：
- 对性能要求极致
- 需要深度系统集成
- 复杂交互动画
- 硬件相关功能开发

💡 面试技巧 ​

答题建议 ​

  1. 先理解问题：确保理解面试官的真实意图
  2. 结构化回答：使用 STAR 法则（情境、任务、行动、结果）
  3. 结合实践经验：用实际项目经验支撑理论
  4. 展示思考过程：即使是难题，也要展示解决思路
  5. 主动延伸：在回答基础上展示更深的理解

准备重点 ​

  • 基础扎实：Java/Kotlin、Android 框架原理
  • 项目经验：准备 2-3 个深度参与的项目
  • 架构思维：能够进行系统设计和技术选型
  • 学习能力：展示对新技术的理解和学习过程
